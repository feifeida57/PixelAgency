# PRD：多智能体协作平台 MVP

## 1. 项目概述

### 1.1 项目名称
**Agency-Agents 多智能体协作平台 MVP**

### 1.2 项目背景
用户希望搭建一个"秘书+团队"的多智能体协作架构，其中：
- **秘书角色**：Claude Code（当前会话的协调者）
- **员工角色**：agency-agents-zh 项目中的 215 个专业智能体

### 1.3 核心目标
基于用户现有的 mimo v2.5 API，实现一个轻量级的多智能体协作系统，让用户能够通过自然语言指令调用不同的专业智能体完成复杂任务。

---

## 2. 用户画像

### 2.1 目标用户
- 拥有 8GB 内存 MacBook Pro 的个人开发者
- 已有 mimo v2.5 API 访问权限
- 希望用 AI 提升工作效率，但本地资源有限

### 2.2 用户痛点
1. **本地内存不足**：无法同时运行多个智能体
2. **上下文理解差**：之前使用 OpenClaw 体验不佳
3. **缺乏协调机制**：不知道如何让多个智能体协作

### 2.3 用户期望
- 轻量级运行，不占用过多本地资源
- 能够调用专业智能体完成特定任务
- 简单易用，无需复杂配置

---

## 3. 功能需求

### 3.1 核心功能（MVP 范围）

#### 3.1.1 智能体加载
**优先级**：P0

**功能描述**：
- 从 agency-agents-zh 项目加载智能体角色定义
- 支持按部门分类加载（工程、设计、营销等）
- 支持单个智能体激活

**验收标准**：
- [ ] 能够读取智能体的 .md 文件
- [ ] 解析智能体的身份、规则、工作流程
- [ ] 在对话中激活指定智能体

#### 3.1.2 自然语言调度
**优先级**：P0

**功能描述**：
- 用户通过自然语言描述任务
- 系统自动选择合适的智能体
- 将任务分配给选中的智能体

**验收标准**：
- [ ] 能够理解用户意图
- [ ] 能够匹配合适的智能体角色
- [ ] 能够生成智能体特定的提示词

#### 3.1.3 结果返回
**优先级**：P0

**功能描述**：
- 智能体完成任务后返回结果
- 结果格式统一（文本、代码、方案等）
- 支持多轮对话

**验收标准**：
- [ ] 能够接收智能体的输出
- [ ] 能够格式化返回给用户
- [ ] 支持追问和细化

### 3.2 扩展功能（后续版本）

#### 3.2.1 多智能体并行
**优先级**：P1

**功能描述**：
- 同时调用多个智能体
- 并行执行独立任务
- 合并多个结果

#### 3.2.2 工作流编排
**优先级**：P2

**功能描述**：
- 定义任务依赖关系
- 自动编排执行顺序
- 支持断点续跑

## 4. 技术方案

### 4.1 架构设计

```
┌─────────────────────────────────────────────────────┐
│                    用户界面层                         │
│              Claude Code（当前会话）                  │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│                   协调调度层                         │
│         Claude Code 智能体协调器                     │
│    ┌─────────────┬─────────────┬─────────────┐      │
│    │  意图识别   │  角色匹配   │  任务分配   │      │
│    └─────────────┴─────────────┴─────────────┘      │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│                   智能体层                           │
│    ┌─────────────┬─────────────┬─────────────┐      │
│    │  工程师     │  设计师     │  营销专家   │      │
│    │  (mimo API) │  (mimo API) │  (mimo API) │      │
│    └─────────────┴─────────────┴─────────────┘      │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│                   数据层                             │
│         agency-agents-zh 智能体定义库                │
│    ┌─────────────┬─────────────┬─────────────┐      │
│    │ engineering │   design    │  marketing  │      │
│    │    *.md     │    *.md     │    *.md     │      │
│    └─────────────┴─────────────┴─────────────┘      │
└─────────────────────────────────────────────────────┘
```

### 4.2 技术栈

| 组件 | 技术选择 | 理由 |
|------|----------|------|
| **协调层** | Claude Code | 用户已有环境，零配置 |
| **智能体运行时** | mimo v2.5 API | 用户已有 API，支持 tool use |
| **智能体定义** | Markdown (.md) | agency-agents-zh 标准格式 |
| **数据存储** | 文件系统 | 简单可靠，无需数据库 |

### 4.3 关键流程

#### 流程一：智能体激活
```
用户输入："帮我写一个 React 登录组件"
    ↓
Claude Code 解析意图：需要前端开发
    ↓
匹配智能体：engineering-frontend-developer.md
    ↓
读取智能体定义：身份、规则、工作流程
    ↓
生成提示词：结合用户需求和智能体定义
    ↓
调用 mimo API：执行任务
    ↓
返回结果：代码 + 说明
```

#### 流程二：多角色协作
```
用户输入："帮我做一个电商网站的 MVP"
    ↓
Claude Code 解析：需要多个角色
    ↓
选择角色：
├── 前端工程师（React）
├── 后端架构师（API）
├── UI 设计师（界面）
└── 产品经理（需求）
    ↓
分配任务：
├── 前端：实现页面
├── 后端：设计 API
├── 设计：提供规范
└── 产品：定义功能
    ↓
并行执行（通过 mimo API）
    ↓
合并结果：返回完整方案
```

## 6. 数据结构

### 6.1 智能体定义文件
```markdown
# 工程师-前端开发者

## 身份
你是高级前端开发工程师，专精 React/Vue 等现代框架。

## 关键规则
1. 代码必须符合 ESLint 规范
2. 组件必须支持 TypeScript
3. 必须编写单元测试

## 工作流程
1. 理解需求
2. 设计组件结构
3. 实现核心逻辑
4. 编写测试
5. 输出文档

## 交付物
- 完整的组件代码
- 单元测试
- 使用文档
```

### 6.2 任务请求格式
```json
{
  "task": "写一个 React 登录组件",
  "role": "engineering-frontend-developer",
  "context": {
    "framework": "React",
    "language": "TypeScript",
    "requirements": ["表单验证", "错误处理", "响应式"]
  }
}
```

### 6.3 任务响应格式
```json
{
  "status": "success",
  "agent": "engineering-frontend-developer",
  "result": {
    "code": "...",
    "tests": "...",
    "docs": "..."
  },
  "metadata": {
    "duration": "2.3s",
    "tokens": 1500
  }
}
```

## 7. 风险评估

### 7.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| mimo API 不稳定 | 高 | 中 | 实现重试机制，缓存结果 |
| 智能体定义质量参差 | 中 | 高 | 建立质量检查机制 |
| 内存不足 | 高 | 低 | 使用云端模式，限制并发 |

### 7.2 产品风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 用户期望过高 | 中 | 中 | 明确 MVP 范围，管理预期 |
| 学习成本高 | 中 | 低 | 提供示例和文档 |
| 场景覆盖不全 | 低 | 高 | 持续迭代，收集反馈 |
